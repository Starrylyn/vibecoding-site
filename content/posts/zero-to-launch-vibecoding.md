## 背景：一个真实的痛点

我每天都在用飞书多维表格管理项目，但需要定期把数据导出成 PDF 报告发给客户。手动操作太繁琐了。

传统的做法：学飞书 API → 搭后端 → 写前端 → 接 PDF 生成库 → 部署。保守估计两周。

我决定用纯 vibecoding 的方式来挑战：7 天上线。

---

## Day 1-2：Vibe 出原型

第一天的目标只有一个：让东西跑起来。

我用了最简单的 prompt：

> "Next.js App Router，一个页面输入飞书多维表格的 app_token 和 table_id，点击后拉取数据并展示为表格，用 Tailwind CSS"

出来的代码有 bug——API 调用的参数格式不对。但我没有去查文档，而是直接把飞书 API 的文档链接扔给了 AI：

> "这里是飞书 API 文档，请按照这个格式修改请求参数。"

10 分钟后，数据显示在了页面上。

---

## Day 3-4：迭代打磨

这两天我发现了一个规律：**vibecoding 最有效的策略是"增量对话"。**

不要一次给太多需求。而是：
1. 描述一个功能 → 看结果
2. 指出不满意的地方 → 看修改
3. 添加下一个功能 → 重复

比如 PDF 导出功能，我的对话是这样的：

- Round 1："加一个导出 PDF 按钮"
- Round 2："PDF 的中文字体是乱码，需要嵌入中文字体"
- Round 3："PDF 加上页眉和页脚，包含公司 logo"
- Round 4："导出前先显示预览"

每一轮 2-5 分钟，叠加起来的效果远超我手写的速度。

---

## Day 5-6：遇到瓶颈

最大的挑战出现在部署阶段。

Vercel 部署时报了一个 Node.js runtime 相关的错误。AI 给的建议在本地可行，部署到 Vercel 就炸。

这时候 vibecoding 的局限性暴露了：**AI 不了解你的具体运行环境。**

解决方案出乎意料地简单：把报错信息完整地复制给 AI，加上"这是 Vercel 的构建日志，请诊断"。AI 识别出是 Node.js 版本不匹配的问题，并给出了正确的 next.config.js 配置。

---

## Day 7：上线

最后一天做了三件事：

1. 加了一个简单的 landing page
2. 配置了 Vercel Analytics
3. 发了 Product Hunt

让我惊讶的是，AI 帮我在 15 分钟内写完了一篇 Product Hunt 的 launch 文案——包括标题方案、tagline 选项和完整的产品描述。

## 复盘：Vibecoding 的得与失

### ✅ 擅长的领域
- **重复性代码**：CRUD、表单、API 集成，大幅提效
- **快速原型**：从 0 到 1 的速度远超传统开发
- **不熟悉的框架**：学新框架的同时直接产出
- **文案和文档**：README、changelog、产品文案

### ❌ 需要警惕的陷阱
- **过度依赖**：不阅读生成的代码会积累技术债务
- **幻觉风险**：API 参数、版本号、配置项可能错误
- **上下文丢失**：长对话后 AI 可能忘记早期的约定
- **环境差异**：本地 OK ≠ 生产 OK

---

## 核心教训

Vibecoding 不是让你的手离开键盘，而是让你的大脑更专注于**判断和决策**。

你不是在放弃编程，你是在升级你的编程抽象层级。

代码依然需要你阅读、理解和维护。但那些曾经需要翻文档、查 Stack Overflow、写样板代码的时间，现在可以用在更重要的事情上——思考你的产品到底要解决什么问题。
